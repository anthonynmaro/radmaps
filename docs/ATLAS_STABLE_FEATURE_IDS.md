# Atlas Stable Feature IDs — design (E2)

Date: June 10, 2026. Part of the editor-v2 epic (docs/STYLE_SYSTEM_EVOLUTION.md). Status: DESIGN — implementation lands in the atlas build before E5 (map_element_overrides) can ship.

## Problem

Map-element overrides (hide/rename/recolor a specific place label or POI) must be keyed so they survive (a) zoom changes, (b) tile rebuilds from newer OSM data, (c) atlas version bumps. Today the pipeline (Geofabrik PBF → Planetiler → PMTiles v3 → R2, immutable versioned manifests) emits no feature ids: properties are semantic only (name/class/rank), no promoteId anywhere, and Planetiler renumbers internal ids on rebuild.

Why tiles must carry the id: MapLibre filters cannot access feature geometry, and feature-state cannot drive layout/text/visibility. Per-feature hide/rename therefore needs a matchable property in the tile. Client-side derivation cannot work for *applying* overrides — only for *capturing* them.

## Design: content-derived `rm_id`

At tile build, for label-bearing layers (`place`, `poi`, `water_name`, `transportation_name`), emit:

```
rm_id = base36( xxhash64( layer | class | normalize(name) | geohash6(labelPoint) ) )
```

- `normalize(name)`: NFKC, casefold, trim, collapse whitespace. Falls back to `ref` then empty.
- `labelPoint`: point features as-is; polygons/lines use Planetiler's computed label centroid, quantized via geohash-6 (~1.2km cell) for places, geohash-7 (~150m) for POIs. Coarse enough to tolerate upstream coordinate nudges, fine enough to split same-name duplicates across a city.
- Deterministic by construction: identical content → identical id across rebuilds and zoom levels. A renamed or relocated-beyond-cell feature gets a new id — correct behavior, since the thing the user overrode no longer exists; the orphaned override is preserved-and-ignored (same posture as effectiveStyleConfig).
- Collisions: 64-bit hash makes accidental collision negligible. Same-name+class within one geohash cell (two co-located chain stores) intentionally share an id — an override hits both; acceptable edge.

### Implementation point

Custom Planetiler profile post-process (the build already runs Dockerized Planetiler): add the property only to the four label layers — negligible tile-size cost, no geometry layers touched. Local build script's existing `promoteId: 'id'` is replaced by `promoteId: 'rm_id'` in the style source spec so MapLibre feature ids resolve to it (enables hover/selected feature-state for the editor highlight too).

### Manifest + capability gating

- Tileset manifest gains `feature_id_scheme: "rm_id@1"`. mapToolCatalog/styleLayerGraph expose a `per-feature-overrides` capability ONLY when the active tileset advertises the scheme — old tiles degrade gracefully (selection highlights nothing override-keyed; no dead controls), per the no-fake-controls rule.
- Scheme version joins the render hash alongside `map_element_overrides` (E5) and `THEME_FIELD_OWNERSHIP_VERSION`.

### Consumption (E5 preview)

- Hide: layer filter `["!", ["in", ["get","rm_id"], ["literal", hiddenIds]]]` composed by buildMapStyle — renders identically in editor/proof/final by construction.
- Rename: `text-field` becomes `["coalesce", ["get", concat("rm_rename_", rm_id)? ]]` — NO. Renames instead use a match expression: `["match", ["get","rm_id"], id1, "New Name", ..., ["get","name"]]`, capped (~100 renames) to keep style JSON sane; cap enforced in the editor.
- Capture: click → queryRenderedFeatures → read `rm_id` directly. No client-side hashing.

## Out of scope / adjacent decisions

- **Contour/elevation labels**: client-generated by maplibre-contour — no persistent identity exists. Per-label persistent hide is NOT offered; the element toolbar gets global size/density plus per-elevation hide ("hide all 2,400 ft labels"), keyed by elevation value — stable by definition. Recorded as a product constraint.
- **Pins/route/segments**: app-owned data with existing ids; nothing needed.
- **Raster presets**: no features, no ids; capability-gated off (unchanged).

## Rollout

1. Planetiler profile change + manifest field; rebuild ONE region (current test region) and verify rm_id stability across two consecutive builds from different-date OSM extracts (the actual stability test).
2. Wire promoteId + capability flag (no UI yet).
3. Full coverage rebuild per the existing coverage plan cadence — overrides feature gates on per-tileset capability, so partial coverage is fine during rollout.
4. E5 consumes.
